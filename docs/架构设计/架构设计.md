## 1. 项目概况

| 维度 | 数据 |
|------|------|
| 类型 | 挂机 RPG，元素战斗系统 |
| 规模 | 58 个 Lua 文件，~28,000 行代码 |
| 引擎 | UrhoX (Lua 5.4) |
| 渲染 | NanoVG 2D 矢量绘图 |
| 存储 | 云端 clientCloud + 本地文件双写 |
| 联网 | 排行榜、世界 Boss、云存档 |

项目采用单线程架构，所有逻辑在 `HandleUpdate` 主循环中驱动。代码按功能分为四层：

```
main.lua (入口 & UI 编排)
  ├── state/     → 数据模型层 (GameState + Install 子模块)
  ├── battle/    → 战斗逻辑层 (15 个子系统)
  ├── view/      → 渲染层 (NanoVG 绘图)
  └── ui/        → UI 组件层 (20 个面板)
```

## 2. 核心架构模式

### 2.1 Install 注入模式

**问题**：GameState 是全局中央状态对象，所有子系统都依赖它。如果把装备、属性计算、战斗状态等方法全写在 GameState.lua 里，文件会膨胀到不可维护。

**方案**：`state/` 目录下的子模块通过 `Install(GameState)` 模式，在初始化阶段将方法注入 GameState 表。

```lua
-- state/Equipment.lua
local Equipment = {}

function Equipment.Install(GameState)
    local Config = require("Config")

    function GameState.AddStone(amount)
        GameState.materials.stone = GameState.materials.stone + amount
    end

    function GameState.EquipItem(slotId, item)
        -- 装备逻辑...
    end

    function GameState.CreateEquip(qualityIdx, chapter, slotId, forceSetId)
        -- 装备生成逻辑...
    end
end

return Equipment
```

```lua
-- GameState.lua 初始化时
require("state.Equipment").Install(GameState)
require("state.StatCalc").Install(GameState)
require("state.Combat").Install(GameState)
```

**优势**：
- GameState 保持为单一入口，外部调用 `GameState.EquipItem()` 无需关心方法定义在哪个文件
- 各子模块可独立编辑，不会产生合并冲突
- 利用闭包捕获 GameState 引用，避免循环 require

**适用场景**：中央数据对象方法过多时（100+ 方法），按职责拆分到子模块并注入。

### 2.2 Proxy 代理层

**问题**：存档系统从单 key (`SaveSystem.lua`) 重构为多槽位分片存储 (`SlotSaveSystem.lua`)。项目中有 20+ 个文件通过 `require("SaveSystem")` 调用存档方法。逐个修改引用路径既费时又容易遗漏。

**方案**：保留 `SaveSystem.lua` 作为薄代理层（~110 行），将所有调用转发给 `SlotSaveSystem`。

```lua
-- SaveSystem.lua (代理层)
local SlotSaveSystem = require("SlotSaveSystem")
local SaveSystem = {}

function SaveSystem.Save()
    SlotSaveSystem.Save()
end

function SaveSystem.SaveNow()
    SlotSaveSystem.SaveNow()
end

function SaveSystem.MarkDirty()
    SlotSaveSystem.MarkDirty()
end

-- Lua 不支持属性拦截，用每帧同步方案
function SaveSystem.SyncFields()
    SaveSystem.offlineSeconds = SlotSaveSystem.offlineSeconds
    SaveSystem._saveCount     = SlotSaveSystem._saveCount
end

return SaveSystem
```

**优势**：
- 零修改迁移：20+ 个使用方不需要改一行代码
- 向后兼容层可以随时删除——当所有引用都迁移到新模块后
- 属性同步问题通过 `SyncFields()` 显式解决，不依赖元表黑魔法

**注意**：Lua table 没有 `__newindex` 拦截器（或者说使用它会引入复杂性），因此属性代理需要额外的同步机制。在这个项目中，`main.lua` 每帧调用 `SaveSystem.SyncFields()` 来保持一致性。

### 2.3 分片云存档系统

**问题**：云存储 API（clientCloud）每个 key 限制 10KB。随着玩家背包装备增长，单 key 存档轻松超限。

**方案**：将存档数据按功能分组，每组独立编码并支持自动分片。

```
分片格式:
  s_N_head     → 索引 (format/version/timestamp/校验码)
  s_N_core     → 玩家基础/关卡/记录
  s_N_currency → 材料/扩展槽
  s_N_equip    → 已装备的装备
  s_N_inv      → 背包 (超 9KB 自动拆为 s_N_inv_0, s_N_inv_1, ...)
  s_N_skills   → 技能/药水
  s_N_misc     → 杂项 (兑换码/称号/日常等)
```

**关键设计细节**：

**(1) 装备数据压缩**

写入时压缩键名（`slot→s`, `qualityIdx→q`, `tier→t`），删除可推导字段（`slotName`, `qualityName`, `qualityColor`, `tierMul`），读取时自动还原。

```lua
-- 压缩: 写入存档时
local function CompressItem(item)
    return { s = item.slot, q = item.qualityIdx, t = item.tier,
             ms = item.mainStat, mv = item.mainValue, ... }
end

-- 解压: 读取存档时 (从 Config 恢复推导字段)
local function DecompressItem(c)
    local item = { slot = c.s, qualityIdx = c.q, tier = c.t, ... }
    item.slotName = GetSlotNameMap()[item.slot]
    item.tierMul  = Config.GetChapterTier(item.tier)
    return item
end
```

**(2) DJB2 校验码**

每个分组写入时计算 DJB2 哈希，存入 head.keys。读取时校验，防止云端数据损坏。

```lua
local function CalcChecksum(str)
    local hash = 5381
    for i = 1, #str do
        hash = ((hash << 5) + hash + string.byte(str, i)) & 0xFFFFFFFF
    end
    return hash
end
```

**(3) 向后兼容**

每次保存同时回写旧格式的 `save_data` 单 key，确保旧版本客户端仍能读取。首次启动时检测旧格式并自动迁移到分片格式。

**(4) 版本迁移链**

```lua
local MIGRATIONS = {
    [1] = function(data) ... data.version = 2; return data end,
    [2] = function(data) ... data.version = 3; return data end,
    -- ...
    [5] = function(data) ... data.version = 6; return data end,
}

local function MigrateData(data)
    local v = data.version or 1
    while MIGRATIONS[v] do
        data = MIGRATIONS[v](data)
        v = data.version
    end
    return data
end
```

迁移前自动备份 (`pre_migrate_vN_backup.json`)，链式执行直到最新版本。

**适用场景**：任何有 KV 大小限制的云存储（如 10KB/key），需要在客户端做数据分片和压缩。

### 2.4 薄编排器模式

**问题**：战斗系统有 15 个子系统（刷怪、AI、弹道、元素反应、Buff、Boss 技能...），如何协调它们的执行顺序？

**方案**：`BattleSystem.lua` 作为"薄编排器"，不包含具体逻辑，只负责按固定顺序调用各子系统的 `Update(dt, bs)` 方法。

```lua
function BattleSystem.Update(dt)
    local bs = BattleSystem
    -- 生存系统 tick
    GameState.UpdateLifeStealTimer(dt)
    GameState.TickHPRegen(dt)
    -- 生成 → AI → 攻击 → 技能 → 弹道 → 敌人 → Buff → 掉落 → 粒子
    Spawner.Update(dt, bs.enemies, bs.areaW, bs.areaH)
    PlayerAI.Update(dt, bs.playerBattle, bs.enemies, ...)
    BuffManager.UpdateSwiftHunter(dt)
    -- ... 20+ 子系统调用
    EnemySystem.UpdateEnemyAbilities(dt, bs)
    ThreatSystem.Update(dt, bs)
    BossSkillTemplates.Update(dt, bs)
    Loot.Update(dt, bs.loots, ...)
    Particles.Update(dt, bs.particles)
    BattleSystem.CleanupDead()
    StageManager.CheckWaveComplete(bs)
end
```

**优势**：
- 执行顺序一目了然，便于调试时序问题
- 各子系统独立开发测试，只需遵守 `Update(dt, bs)` 接口
- BattleSystem 本身极薄（~300 行），大量逻辑分散在子模块中

**关键约定**：所有子系统通过 `bs` 参数访问共享状态（enemies, bullets, loots 等），而不是直接 require BattleSystem，避免循环依赖。

### 2.5 Boss 技能三层模板

**问题**：随着章节增加（14 章+），每个 Boss 都有独特的技能组合。如果为每个 Boss 硬编码技能逻辑，代码量会线性增长且难以维护。

**方案**：将 Boss 技能抽象为三层正交架构。

```
模板层 (BossSkillTemplates.lua)  → 行为逻辑, 元素无关
  ↓
参数层 (StageConfig phases)      → 数值配置, 章节特调
  ↓
渲染层 (DrawBossTemplates.lua)   → NanoVG 程序化视觉
```

**15 个可复用模板**：

| 类别 | 模板 | 行为 |
|------|------|------|
| ATK | barrage, breath, pulse, spikes, detonate | 输出型技能 |
| DEF | armor, crystal, shield, regen | 防御型技能 |
| CTL | field, barrier, decay, vortex | 控制型技能 |
| SUM | minion, guard | 召唤型技能 |

**模板注册机制**：

```lua
BossSkillTemplates.RegisterTemplate("ATK_barrage", {
    init   = function(bs, enemy, cfg) ... end,
    update = function(dt, bs, enemy, cfg) ... end,
})
```

**配置桥接**：`flattenSkillCfg()` 将 StageConfig 的嵌套配置展平为模板期望的扁平结构，处理字段重命名和语义映射。

```lua
-- StageConfig 中的 Boss 阶段配置
phases = {
    { triggerHp = 1.0, skills = {
        { template = "ATK_barrage", element = "poison",
          params = { count = 8, damage = 1.2, interval = 3.0 } },
        { template = "DEF_crystal", element = "poison",
          params = { hp = 50000, count = 2, onCoreTick = { ... } } },
    }},
    { triggerHp = 0.5, skills = { ... } },  -- 第二阶段
}
```

**优势**：
- 新增 Boss 只需配置 phases 表，不写新逻辑代码
- 同一模板可应用不同元素和数值，组合出丰富变化
- 渲染层独立，视觉效果可以跨模板复用

### 2.6 怪物组装工厂

**问题**：14 个章节、每章 10 关、数百种怪物组合——如何避免手动配置每一种怪物？

**方案**：MonsterTemplates 使用三层正交设计，通过 `Assemble()` 函数动态组装。

```
行为模板 (7 种)  × 抗性模板 (12 种) × 章节主题 (14 套) = 1,176 种组合
```

```lua
M.Behaviors = {
    swarm    = { hp=35, atk=3, def=0, speed=55, ... },
    tank     = { hp=160, atk=9, def=14, speed=15, ... },
    glass    = { hp=20, atk=14, def=0, speed=75, ... },
    -- ...
}

M.Resists = {
    balanced  = { fire=10, ice=10, poison=10, ... },
    fire_res  = { fire=50, ice=5, poison=10, ... },
    -- ...
}
```

每个行为模板还定义了 `defaultTags`（始终携带的能力）和 `optionalTags`（按章节解锁的能力），实现了能力的渐进解锁。

```lua
tank = {
    defaultTags  = { antiHeal = true },            -- 始终有
    optionalTags = { hpRegen = 2, sporeCloud = 7 }, -- 第2章起有回血, 第7章起有孢子云
}
```

**优势**：
- 新增章节只需定义主题和贴图，不需要手写怪物数值
- 正交组合保证了多样性，同时避免了组合爆炸的维护成本
- 能力标签有 1~5 级参数表，数值精确可控

### 2.7 滑动窗口 DPS 追踪

**问题**：需要实时显示 DPS（每秒伤害），同时支持世界 Boss 模式的会话伤害统计。

**方案**：`DamageTracker.lua` 使用固定时间桶的滑动窗口算法。

```lua
local DPS_WINDOW     = 5.0    -- 统计窗口 5 秒
local DPS_BUCKET_DUR = 0.5    -- 每桶 0.5 秒
local DPS_BUCKET_CNT = 10     -- 最多 10 个桶

function DamageTracker.Record(amount)
    totalDamage_ = totalDamage_ + amount
    currentBucket_.dmg = currentBucket_.dmg + amount
end

function DamageTracker.Update(dt)
    currentBucket_.elapsed = currentBucket_.elapsed + dt
    if currentBucket_.elapsed >= DPS_BUCKET_DUR then
        table.insert(dpsBuckets_, currentBucket_)
        currentBucket_ = { dmg = 0, elapsed = 0 }
        while #dpsBuckets_ > DPS_BUCKET_CNT do
            table.remove(dpsBuckets_, 1)
        end
    end
end

function DamageTracker.GetRealtimeDPS()
    local totalDmg, totalTime = currentBucket_.dmg, currentBucket_.elapsed
    for _, b in ipairs(dpsBuckets_) do
        totalDmg = totalDmg + b.dmg
        totalTime = totalTime + b.elapsed
    end
    return totalTime < 0.1 and 0 or math.floor(totalDmg / totalTime)
end
```

模块还包含一个巧妙的 **Boss 血条分层系统**：用指数递增的血量层（每层 ×1.5），通过对数反推当前层号，实现了无限血量下的直观进度显示。

## 3. 数据流与生命周期

### 启动流程

```
main.lua Start()
  → GameState.Init()          -- 初始化中央状态
  → Install 子模块注入         -- Equipment/StatCalc/Combat
  → SlotSaveSystem.Init()     -- 加载云端 meta (异步)
  → StartScreen 显示           -- 用户选择存档槽
  → SlotSaveSystem.LoadSlot() -- 加载选中槽位 (异步)
  → Deserialize + Migrate     -- 反序列化 + 版本迁移
  → BuildGameUI()             -- 构建游戏 UI
  → VersionReward.Check()     -- 版本更新奖励
  → OfflineChest.Check()      -- 离线奖励
```

### 战斗循环

```
HandleUpdate(dt)
  → SlotSaveSystem.Update(dt)   -- 自动存档计时
  → BattleSystem.Update(dt)     -- 战斗编排 (调用 15 个子系统)
  → UI 刷新 (节流 0.3s)         -- HUD/StatusBars/TabBar
```

### 存档流程

```
MarkDirty() → 3秒延迟 → Save()
  → Serialize()                  -- GameState → table
  → SaveLocal()                  -- 原子写入本地 (先 .tmp 后 rename)
  → DoCloudSave()                -- 分片编码 + BatchSet 异步上传
    → SplitIntoGroups()          -- 按功能分组
    → CompressEquipmentTable()   -- 装备键名压缩
    → EncodeChunkedGroups()      -- 分片 + DJB2 校验
    → BatchSet + 向后兼容回写    -- 新旧格式双写
```

## 4. 值得复用的设计

### 4.1 中央状态 + Install 注入

**复用价值：★★★★★**

适合任何存在"上帝对象"（God Object）的 Lua 项目。与面向对象的继承不同，Install 模式用闭包替代类继承，更符合 Lua 的动态特性。

**要点**：
- 子模块通过闭包捕获 GameState 引用
- 外部调用者完全不知道方法定义在哪个文件
- 可以按领域（装备、属性、战斗）独立开发和测试

### 4.2 分片存储 + 装备压缩

**复用价值：★★★★★**

任何使用 KV 云存储的游戏都会遇到大小限制问题。这套方案的核心思路：
- 按功能分组，而非机械分割
- 压缩可推导字段，减少数据量
- 自动分片 + 校验码，保证可靠性
- 双写兼容旧版本

### 4.3 版本迁移链

**复用价值：★★★★☆**

`MIGRATIONS[v]` 链式迁移是成熟的模式。关键是迁移前备份和迁移后的严格校验。

### 4.4 Proxy 代理层

**复用价值：★★★★☆**

任何大型重构中，保留旧接口名的转发层可以大幅降低风险。在确认所有调用方都已迁移后，可以安全删除。

### 4.5 三层 Boss 模板

**复用价值：★★★★☆**

适合 Boss/敌人种类多、技能需要复用的游戏。模板层处理通用行为，参数层控制数值差异，渲染层独立演化。

### 4.6 DPS 滑动窗口

**复用价值：★★★☆☆**

简洁高效的实时统计方案。固定桶大小避免了复杂的环形缓冲区，对 Lua GC 友好。

## 5. 不足与改进方向

### 5.1 StageConfig 巨型文件（3,243 行）

**现状**：所有 14 个章节的关卡配置、Boss 定义、怪物组合、对话文本都集中在一个文件中。

**问题**：
- 查找特定章节内容需要大量滚动
- 多人协作时容易冲突
- AI 辅助开发时上下文窗口被占满

**改进方案**：按章节拆分为独立文件，用注册模式汇总。

```lua
-- StageConfig.lua (注册中心)
local StageConfig = {}
local chapters = {}

function StageConfig.RegisterChapter(ch, data)
    chapters[ch] = data
end

function StageConfig.GetStage(ch, stage)
    return chapters[ch] and chapters[ch].stages[stage]
end

-- 自动加载所有章节
for i = 1, 14 do
    require("chapters.ch" .. i)  -- 每个文件调用 RegisterChapter
end

return StageConfig
```

```lua
-- chapters/ch13.lua
local StageConfig = require("StageConfig")
StageConfig.RegisterChapter(13, {
    name = "极寒王庭",
    stages = { ... },
    bosses = { ... },
})
```

**收益**：每个章节文件 ~200-300 行，职责清晰，可以独立编辑。

### 5.2 GameState 承载过多运行时状态

**现状**：GameState 除了持久化数据（等级、装备、技能），还承载大量战斗运行时状态（各种 Buff/Debuff 计时器、套装效果变量）。这些变量以 `GameState._` 前缀存放。

```lua
-- 运行时套装状态 (不持久化，每场战斗重置)
GameState._swiftHunterTarget = nil
GameState._swiftHunterStacks = 0
GameState._fissionEnergy = 0
GameState._shadowStacks = 0
GameState._dragonPhase = nil
-- ... 40+ 个运行时变量
```

**问题**：
- 持久化数据和运行时数据混在一起，Serialize() 需要显式排除
- 新增套装时容易忘记在 BattleSystem.Init() 中重置

**改进方案**：将运行时状态抽取为独立的 `BattleState` 对象，在 `BattleSystem.Init()` 时整体创建。

```lua
-- BattleState.lua
local function CreateBattleState()
    return {
        swiftHunter = { target = nil, stacks = 0, stormTimer = 0 },
        fission     = { energy = 0, atkSpdTimer = 0 },
        shadow      = { stacks = 0, postBurstTimer = 0 },
        dragon      = { phase = nil, hitCount = 0 },
        -- ...
    }
end
```

**收益**：运行时状态与持久化数据彻底分离，整体创建/销毁，无遗漏风险。

### 5.3 缩放曲线的参数化不足

**现状**：`tierMul = 1 + 99 × ln(ch) / ln(100)` 的对数曲线在高章节（9-12）趋于平坦，导致 3 个章节的装备差距仅 12.8%。`PLAYER_POWER_MUL` 用查表+手动调参来补偿，已经修改过多次。

**问题**：
- 对数曲线的平坦段使得高章节装备"手感"差异不明显
- PLAYER_POWER_MUL 查表值需要反复手调，缺乏理论锚定

**改进方案**：采用分段公式或参数化曲线，让每个章节段的增长率可独立控制。

```lua
-- 分段线性 + 指数混合
local function GetTierMul(ch)
    if ch <= 4 then
        return 1 + (ch - 1) * 12    -- 前期快速成长
    elseif ch <= 8 then
        return 37 + (ch - 4) * 8    -- 中期稳定成长
    else
        return 69 + (ch - 8) * 5    -- 后期缓慢但可感知
    end
end
```

**收益**：每段增长率可独立调节，不再依赖对数函数的固有形状。

### 5.4 Config.lua 的职责过重

**现状**：Config.lua (~1,585 行) 承载了数值配置、公式定义、装备属性、套装定义、掉落模板、图标路径等多种职责。

**改进方案**：按主题拆分为子配置模块。

```
Config/
  ├── Base.lua       → 基础常量 (重力、速度、经验公式)
  ├── Equipment.lua  → 装备品质、词条、升级公式
  ├── Sets.lua       → 套装定义
  ├── Drop.lua       → 掉落模板、金币缩放
  └── Assets.lua     → 图标路径、贴图映射
```

Config.lua 作为聚合入口，将所有子模块的内容合并导出，保持外部调用兼容。

### 5.5 BuffManager 的子模块重导出

**现状**：BuffManager.lua 通过 `for k, v in pairs(NewSets) do BuffManager[k] = v end` 将子模块的所有函数复制到自身。

```lua
local NewSets  = require("battle.buffs.NewSets")
local T13Sets  = require("battle.buffs.T13Sets")

for k, v in pairs(NewSets)  do BuffManager[k] = v end
for k, v in pairs(T13Sets)  do BuffManager[k] = v end
```

**问题**：
- 命名冲突风险：两个子模块如果有同名函数，后者会覆盖前者
- 调试时 `BuffManager.UpdateSwiftHunter` 的实际定义在 `NewSets.lua` 中，增加了溯源成本

**改进方案**：使用命名空间或显式导出。

```lua
-- 方案 A: 命名空间
BuffManager.sets   = NewSets
BuffManager.t13    = T13Sets
-- 调用: BuffManager.sets.UpdateSwiftHunter(dt)

-- 方案 B: 显式列出导出函数 (更安全)
BuffManager.UpdateSwiftHunter = NewSets.UpdateSwiftHunter
BuffManager.UpdateFissionForce = NewSets.UpdateFissionForce
-- ...
```

### 5.6 cjson 稀疏数组陷阱

**现状**：项目曾因 Lua table 的数字索引（如 `claimed[1]=true, claimed[3]=true`）在 cjson 序列化时被处理为稀疏 JSON 数组（`[true, null, true]`），反序列化后丢失数据。

**已修复**：改用 `tostring(idx)` 作为 key，确保 cjson 生成对象而非数组。

**教训**：在 Lua + cjson 环境中，**任何可能存在间隔的数字索引 table 都应使用字符串 key**。建议在项目中建立编码规范：

```lua
-- ❌ 危险: 稀疏数字索引 → cjson 可能生成有 null 的数组
claimed[questIdx] = true

-- ✅ 安全: 字符串 key → cjson 始终生成对象
claimed[tostring(questIdx)] = true
```

## 6. 总结

### 做得好的

| 模式 | 核心思想 | 一句话总结 |
|------|---------|-----------|
| Install 注入 | 闭包替代继承 | 拆分上帝对象，保持统一入口 |
| Proxy 代理 | 接口不变，实现替换 | 零成本大型重构 |
| 分片存档 | 按功能分组 + 压缩 + 校验 | 10KB 限制下的可靠云存储 |
| 薄编排器 | 只排序，不处理 | 战斗循环清晰可维护 |
| Boss 三层模板 | 正交分离 | 15 个模板组合出无限 Boss |
| 怪物组装工厂 | 行为 × 抗性 × 主题 | 1,176 种组合零手动配置 |

### 需要改进的

| 问题 | 根因 | 优先级 |
|------|------|--------|
| StageConfig 3,243 行 | 所有章节集中在一个文件 | 高 |
| Config.lua 1,585 行 | 职责过多 | 中 |
| 运行时状态污染 GameState | 缺少独立的战斗状态容器 | 中 |
| 缩放曲线平坦段 | 对数函数固有形状 | 低（已通过查表缓解） |
| BuffManager 批量复制 | 命名冲突风险 | 低 |

**核心原则**：在 Lua 这种动态语言中，好的架构不依赖类型系统来保证正确性，而是依赖**清晰的约定**（Install 接口）、**薄的中间层**（Proxy 代理、编排器）和**正交的组合**（模板 × 参数 × 渲染）。
