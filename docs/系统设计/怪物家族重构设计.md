# 怪物家族重构设计

> 版本: v1.1
> 日期: 2026-03-25
> 状态: 设计阶段
> 依赖: MonsterTemplates.lua (行为模板层保留), StageConfig.lua (关卡层重构)

---

## 一、痛点与目标

### 当前痛点

1. **新增章节成本极高**: 每章需要手写 6-8 个怪物定义（每个 10-20 行）+ 10 关 waves 配置 + 2 个 Boss + 贴图资源 = StageConfig 3950 行且持续膨胀
2. **怪物没有身份**: "灰烬鼠"和"霜魔小鬼"在代码层面是完全独立的定义，玩家也感知不到它们都是"蜂群型"怪物
3. **缩放全靠数值**: 章节之间的差异只有 `scaleMul` 数值放大 + 换贴图，没有机制层面的变化

### 设计目标

1. **怪物家族复用**: 定义一次家族，所有章节通过 scaleMul + 换皮复用，新增章节只需指定"本章出现哪些家族"
2. **StageConfig 行数降低 70%+**: 从 3950 行降至 ~800-1200 行
3. **保持战斗体验不变**: 重构是纯架构改造，玩家感知不到变化
4. **为后续系统铺路**: IP（Item Power）系统需要怪物等级概念，家族系统自然提供

## 二、核心概念：暗黑4怪物家族 vs 当前系统

### 暗黑4的怪物家族

暗黑4中，怪物按**家族（Monster Family）**分组：

| 家族 | 成员 | 出现区域 |
|------|------|---------|
| 骷髅 | 骷髅战士、骷髅弓手、骷髅法师、骷髅队长 | 墓地、地下城 |
| 蜘蛛 | 小蜘蛛（swarm）、毒蜘蛛（debuffer）、蛛后（boss） | 洞穴、密林 |
| 坠魔 | 坠魔小鬼（swarm）、坠魔巫师（caster）、坠魔首领 | 地狱裂隙 |

关键特征：
- **家族内成员共享视觉主题**（骨质/蛛丝/恶魔角）
- **家族绑定区域**，不同区域遇到不同家族组合
- **同一家族在不同难度下数值缩放**，但行为模式不变
- **家族内角色分工明确**：总有 swarm（冲脸）、ranged（远程）、elite（精英）、boss

### 当前系统 vs 家族系统

| 维度 | 当前 | 家族系统 |
|------|------|---------|
| 怪物身份 | `ash_rat` = {name, hp, atk, ...} 扁平定义 | `undead.swarm` = 不死族的蜂群角色 |
| 章节绑定 | 每章手写全部怪物 | 每章指定 1-3 个家族，自动展开 |
| 新增怪物 | 逐个手写属性 | 新增家族一次，所有章节可复用 |
| 视觉主题 | 每个怪物独立贴图路径 | 家族内共享色调，角色间差异靠行为模板 |
| 行为模板 | 有（7种），但和怪物定义是分离的 | 家族成员直接引用行为模板 |

## 三、架构设计

### 三层架构（改造后）

```
行为模板 (MonsterTemplates.Behaviors)     ← 保留不变，7种
    ↑ 引用
怪物家族 (MonsterFamilies)                ← 新增核心层
    ↑ 引用
章节配置 (StageConfig.CHAPTERS)           ← 大幅简化
```

### 数据流

```
章节配置: { families = {"undead", "beast"}, scaleMul = ... }
    ↓
家族展开: undead → 7个成员 (swarm/tank/glass/bruiser/debuffer/caster/exploder)
    ↓
行为模板 + 抗性模板 + 能力标签 → Assemble() → Spawner 兼容定义
    ↓
关卡编排模板: 自动生成 10 关 waves
```

## 四、怪物家族定义

### 4.1 家族结构

```lua
---@class MonsterFamily
---@field id string              家族ID
---@field name string            家族名称（显示用）
---@field theme string           视觉主题描述
---@field element string         家族主属元素
---@field colorBase number[]     基础色调 {r, g, b}
---@field resistProfile string   家族默认抗性模板ID
---@field members table<BehaviorId, FamilyMember>  成员表（按行为模板索引）

---@class FamilyMember
---@field name string            成员名称
---@field image string           贴图路径
---@field color number[]|nil     覆盖色调（nil 则用家族基础色调）
---@field resistId string|nil    覆盖抗性模板（nil 则用家族默认）
---@field tags table|nil         额外能力标签 { tagName = level }
---@field overrides table|nil    属性覆盖 { hp = 50, speed = 60, ... }
```

### 4.2 初始家族列表（8个家族）

参考暗黑4的家族设计，结合当前14章的怪物分布，归纳为 **8 个家族**：

| ID | 名称 | 主属元素 | 设计灵感 | 适用章节 |
|----|------|---------|---------|---------|
| `undead` | 不死族 | arcane | D4骷髅/亡灵 | 1, 4, 10, 12 |
| `beast` | 野兽族 | physical | D4狼人/野兽 | 1, 2, 6, 7 |
| `elemental_fire` | 火元素族 | fire | D4熔岩/火焰 | 1, 3, 11 |
| `elemental_ice` | 冰元素族 | ice | D4冰霜/寒冰 | 2, 5, 13 |
| `elemental_poison` | 毒元素族 | poison | D4瘟疫/腐蚀 | 3, 7, 14 |
| `arcane` | 奥术族 | arcane | D4虚空/暗影 | 4, 8, 12 |
| `divine` | 圣光族 | holy | D4天使/圣骑 | 9 |
| `aquatic` | 深海族 | water | D4深海/潮汐 | 5, 6, 13, 16 |

> 每个家族 7 个成员（对应 7 个行为模板），一次定义，终身复用。

## 五、章节配置简化

### 5.1 新的章节配置结构

```lua
---@class ChapterConfig
---@field name string              章节名称
---@field families string[]        出场家族ID列表 (1-3个)
---@field resistCoeff number       抗性系数
---@field boss table               Boss配置 (保留手工设计)
---@field stageNames string[]|nil  10关名称（可选，有默认生成逻辑）
---@field waveOverrides table|nil  特殊关卡的波次覆盖（可选）
```

### 5.2 章节→家族映射

| 章 | 名称 | 家族组合 | 主家族 | 辅助家族 | 抗性系数 |
|----|------|---------|--------|---------|---------|
| 1 | 灰烬荒原 | beast + elemental_fire | beast | elemental_fire | 0.3 |
| 2 | 冰封深渊 | elemental_ice + beast | elemental_ice | beast | 0.3 |
| 3 | 熔岩炼狱 | elemental_fire + elemental_poison | elemental_fire | elemental_poison | 0.5 |
| 4 | 幽暗墓域 | undead + elemental_poison | undead | elemental_poison | 0.5 |
| 5 | 深海渊域 | aquatic + elemental_poison | aquatic | elemental_poison | 0.7 |
| 6 | 雷鸣荒漠 | arcane + beast | arcane | beast | 0.7 |
| 7 | 瘴毒密林 | elemental_poison + beast | elemental_poison | beast | 0.85 |
| 8 | 虚空裂隙 | arcane + undead | arcane | undead | 0.85 |
| 9 | 天穹圣域 | divine + arcane | divine | arcane | 1.0 |
| 10 | 永夜深渊 | undead + arcane | undead | arcane | 1.0 |
| 11 | 焚天炼狱 | elemental_fire + undead | elemental_fire | undead | 1.1 |
| 12 | 时渊回廊 | arcane + elemental_fire | arcane | elemental_fire | 1.1 |
| 13 | 寒渊冰域 | elemental_ice + aquatic | elemental_ice | aquatic | 1.2 |
| 14 | 腐蚀魔域 | elemental_poison + undead | elemental_poison | undead | 1.2 |

### 5.4 对比：改造前 vs 改造后

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 每章怪物定义 | 6-8个怪物 × 15行 = ~120行 | 0行（引用家族） |
| 每章 waves 配置 | 10关 × 15行 = ~150行 | 自动生成（~0行）或少量 override |
| 每章 Boss 配置 | 2个Boss × 60行 = ~120行 | ~4行（原型三元组引用） |
| 每章总行数 | ~400行 | ~15-20行 |
| 14章总行数 | ~3950行 | ~400-500行（含家族定义200行 + Boss原型200行） |
| 新增一章 | 写怪物+waves+Boss = 半天 | 填一个表 = 5分钟，零美术成本 |

## 六、关卡自动编排

### 6.1 编排模板（10关标准模式）

当前每章10关的编排模式高度一致（见怪物模板系统文档第七节），可以提取为通用模板：

```lua
---@alias StageTemplate { pattern: string, primaryRole: string, description: string }

local STAGE_TEMPLATES = {
    -- 关 1: 纯蜂群入门
    { pattern = "swarm_only",
      waves = {
          { roles = { { role = "swarm", count = 25, family = "primary" } } },
          { roles = { { role = "swarm", count = 30, family = "primary" } } },
      },
    },
    -- 关 2: 引入肉盾
    { pattern = "intro_tank",
      waves = {
          { roles = { { role = "swarm", count = 20, family = "primary" }, { role = "tank", count = 6, family = "primary" } } },
          { roles = { { role = "swarm", count = 15, family = "primary" }, { role = "tank", count = 8, family = "primary" } } },
      },
    },
```

### 6.2 编排函数

```lua
--- 根据章节配置和编排模板，自动生成 10 关的 waves
---@param chapterCfg ChapterConfig
---@param chapter number
---@return table[] stages  10关配置
function StageConfig.GenerateStages(chapterCfg, chapter)
    local families = chapterCfg.families
    local primaryFamily = MonsterFamilies.Get(families[1])
    local secondaryFamily = MonsterFamilies.Get(families[2] or families[1])

    local stages = {}
    for i, template in ipairs(STAGE_TEMPLATES) do
        local stage = { waves = {} }
        for _, waveDef in ipairs(template.waves) do
            local wave = { monsters = {} }
            for _, roleDef in ipairs(waveDef.roles) do
                local family = roleDef.family == "secondary" and secondaryFamily or primaryFamily

                if roleDef.role == "boss_mid" then
                    -- Boss 走原型解析
                    local bossId = BossArchetypes.Resolve(chapterCfg.boss.mid, chapter)
                    table.insert(wave.monsters, { id = bossId, count = 1 })
                elseif roleDef.role == "boss_final" then
                    local bossId = BossArchetypes.Resolve(chapterCfg.boss.final, chapter)
                    table.insert(wave.monsters, { id = bossId, count = 1 })
                else
                    -- 通过家族+行为模板组装怪物
                    local monsterId = family.id .. "_" .. roleDef.role  -- e.g. "undead_swarm"
                    table.insert(wave.monsters, { id = monsterId, count = roleDef.count })
                end
            end
            table.insert(stage.waves, wave)
        end
        table.insert(stages, stage)
    end
    return stages
end
```

### 6.3 怪物解析（Spawner 适配）

Spawner.BuildQueue 不再从 `StageConfig.MONSTERS[id]` 查找扁平定义，而是：

```lua
-- 新的怪物解析流程
local function resolveMonster(monsterId, chapter, tagLevels)
    -- 1. 先查 Boss 表（Boss 仍手工定义）
    if StageConfig.BOSSES[monsterId] then
        return StageConfig.BOSSES[monsterId]
    end

    -- 2. 解析 "familyId_behaviorId" 格式
    local familyId, behaviorId = monsterId:match("^(.+)_(%w+)$")
    local family = MonsterFamilies.Get(familyId)
    local member = family.members[behaviorId]

    -- 3. 通过 MonsterTemplates.Assemble 组装
    return MonsterTemplates.Assemble(behaviorId, member.resistId or family.resistProfile, chapter, {
        name = member.name,
        image = member.image,
        color = member.color or family.colorBase,
        tags = mergeTags(behaviorId, tagLevels),  -- 行为默认标签 + 章节标签等级
    })
end
```

### 7.2 标签自动分配规则

哪个行为模板获得哪个标签，由 `MonsterTemplates.Behaviors[id].optionalTags` 决定（已有机制）：

```
如果章节 tagLevels 声明了某标签
  且 该行为模板的 optionalTags 包含该标签
  且 该标签的引入章节 <= 当前章节

```

**这消除了逐怪手配标签的工作量**。

## 八、Boss 原型系统

### 8.1 现状分析：Boss 已经在事实上复用

审查现有 28 个 Boss（14章 × 中Boss + 终Boss），发现**结构高度重复**：

**旧版 Ch1-12（扁平字段，4 种固定组合）：**

| 类型 | 技能组合 | 使用次数 |
|------|---------|---------|
| 中Boss A | barrage + iceArmor | Ch1/3 |
| 中Boss B | barrage + iceArmor + summon | Ch4/5/6/7/8/9/10/11/12 |
| 终Boss 统一 | dragonBreath + frozenField + iceArmor + iceRegen | 全部 Ch1-12 |

**新版 Ch13-15（phases 模板系统，2 种结构）：**

| 类型 | 阶段结构 | Ch13 | Ch14 | Ch15 |
|------|---------|------|------|------|
| 中Boss | P1(barrage+spikes+SUM) → P2(barrage_360+field+armor+crystal) | 格拉西恩 | 维诺莎 | 伊格尼斯 |
| 终Boss | P1(breath+spikes+SUM_guard) → P2(breath+decay+barrier+field+shield) → P3(armor+regen+vortex+detonate) | 尼弗海姆 | 涅克洛斯 | 萨拉曼德 |

**结论**：Boss 之间的差异几乎只有**元素类型**和**数值参数**，战斗结构完全相同。这和暗黑4的做法一致——应该正式抽象为**原型**。

### 8.2 暗黑4 的 Boss 复用策略

D4 的 Boss 复用分层级：

| 层级 | 做法 | 独特设计量 |
|------|------|----------|
| 精英怪 | 家族成员 + 随机词缀 | 0（全自动） |
| 地城 Boss | 少量原型 × 换元素换皮 | ~10 个原型 |
| 世界 Boss | 固定 3 个轮换 | 3 个 |
| 剧情 Boss | 唯一设计 | ~5 个 |

真正独特设计的 Boss 不超过 15 个，其余几百个 Boss 遭遇都是"原型 × 元素 × 数值"的排列组合。

### 8.4 原型 × 元素 × 家族 = Boss 实例

章节配置中，Boss 变成简单的**三元组引用**：

```lua
-- 章节配置中的 Boss 定义 (~4行)
boss = {
    mid   = { archetype = "striker_2p",   element = "fire", family = "elemental_fire" },
    final = { archetype = "sovereign_3p", element = "fire", family = "elemental_fire" },
},
```

#### 解析流程

```
章节配置: { archetype, element, family }
  ↓
BossArchetypes.Resolve(archetypeId, element, family, chapter)
  ↓
1. Deep-copy 原型的 phases 结构
2. 注入元素 → 决定 Debuff 类型和技能贴图
3. 注入家族 → 决定抗性模板和色调
4. 注入章节数值 → hp/atk/def × scaleMul
5. 输出 Spawner 兼容的 Boss 定义（含 phases）
```

#### 元素决定 Debuff 风味

同一个原型在不同元素下，"打击感"完全不同：

| 元素 | onHit Debuff | decay 目标 | shield 反应 | 玩家体验 |
|------|-------------|-----------|------------|---------|
| ice | slow(减速) | moveSpeed(跑不动) | melt(融化) | 冻得动不了 |
| poison | venomStack(蚀毒叠层) | def(扛不住) | purify(焚净) | 越打越脆 |
| fire | blazeStack(灼烧叠层) | atkSpeed(打不动) | quench(淬灭) | 越打越慢 |
| arcane | silence(沉默) | crit(暴击递减) | dispel(驱散) | 技能废了 |
| water | corrode(腐蚀护甲) | healReduce(回复压制) | evaporate(蒸发) | 回不了血 |
| holy | blind(致盲) | dmgReduce(伤害递减) | judgment(审判) | 打不出伤害 |

#### 元素注入函数

```lua
--- 根据元素类型注入对应的 Debuff 回调
---@param phases table  deep-copied phases
---@param element string  元素类型
local function injectElementFlavor(phases, element)
    local ELEMENT_FLAVOR = {
        ice    = { onHit = { slow = 0.10, slowDuration = 1.5 },    decay = "moveSpeed", shieldReaction = "melt",     weakElement = "fire"  },
        poison = { onHit = "ApplyVenomStackDebuff",                 decay = "def",       shieldReaction = "purify",   weakElement = "fire"  },
        fire   = { onHit = "ApplyBlazeDebuff",                      decay = "atkSpeed",   shieldReaction = "quench",   weakElement = "water" },
        arcane = { onHit = { silence = 1.5 },                       decay = "crit",      shieldReaction = "dispel",   weakElement = "holy"  },
        water  = { onHit = { corrode = 0.03 },                      decay = "healReduce", shieldReaction = "evaporate", weakElement = "fire" },
        holy   = { onHit = { blind = 2.0 },                         decay = "dmgReduce", shieldReaction = "judgment",  weakElement = "arcane" },
    }
    local flavor = ELEMENT_FLAVOR[element]
    -- 遍历所有技能槽，注入 onHit/effect/coreEffect/shield_reaction
    for _, phase in ipairs(phases) do
        for _, skill in ipairs(phase.skills) do
            if skill.template == "ATK_barrage" or skill.template == "ATK_breath" then
                skill.params.onHit = skill.params.onHit or flavor.onHit
            end
            if skill.template == "CTL_decay" then
                skill.params.stat = skill.params.stat or flavor.decay
            end
            if skill.template == "DEF_shield" then
                skill.params.shield_reaction = skill.params.shield_reaction or {
                    weakReaction = flavor.shieldReaction,
                    weakElement = flavor.weakElement,
                    weakMultiplier = 2.5,
                }
            end
            if skill.template == "CTL_field" then
                skill.params.effect = skill.params.effect or flavor.onHit
            end
            if skill.template == "CTL_vortex" then
                skill.params.coreEffect = skill.params.coreEffect or flavor.onHit
            end
        end
    end
end
```

### 8.6 原型组合覆盖率

4 个中Boss原型 × 7 个终Boss原型 × 6 种元素 = **168 种组合**

实际 14 章只需要 28 个 Boss，覆盖率仅 16.7%。即使扩展到 30 章也不会用完。

| 原型 | 适用场景 | 当前映射 |
|------|---------|---------|
| `striker_2p` | 弹幕密集、节奏快的中Boss | Ch13/14/15 中Boss |
| `charger_2p` | 近战压迫、直线威胁的中Boss | Ch1-12 终Boss 的简化版 |
| `summoner_2p` | 召唤流、间接压力的中Boss | 新增，丰富多样性 |
| `fortress_2p` | 耐久型、持久战的中Boss | 旧版 Ch1-12 终Boss 升级 |
| `sovereign_3p` | 标准终Boss，渐进升级 | Ch13/14/15 终Boss |
| `overlord_3p` | 弹幕地狱终Boss | 新增，攻击导向 |
| `herald_3p` | 防御反击终Boss | 新增，防御导向 |

### 8.7 与现有 BossSkillTemplates 的关系

BossSkillTemplates（15 个技能模板）完全不受影响。Boss 原型只是对技能模板的**组合方式**做了归纳：

```
BossSkillTemplates (15个技能模板)   ← 不变，底层能力
    ↑ 组合引用
BossArchetypes (7个原型)            ← 新增，组合层
    ↑ 参数化引用
章节配置 (archetype + element)      ← 简化为 4 行
```

### 8.8 旧版 Boss 迁移

Ch1-12 的旧版扁平 Boss 可以逐步迁移为原型引用：

| 旧 Boss | 映射原型 | 元素 |

| boss_golem (Ch1终) | fortress_2p | arcane |
| boss_ice_witch (Ch2中) | striker_2p | ice |

迁移时 BossSkillTemplates.Update 的路由逻辑会自动处理：有 `phases` 字段走模板系统，无 `phases` 走旧扁平逻辑。可以逐个迁移，不需要一次性改完。

### 9.1 贴图路径规范

```
assets/Textures/mobs/

└── mine/                     ← 保留：折光矿脉专属贴图
```

### 9.2 贴图复用

同一家族的怪物共享色调，通过行为模板的体型差异（radius）和色调微调区分。这意味着：

- **不需要为每章每怪单独画贴图**: 8 家族 × 7 行为 = 56 张怪物贴图（当前有 100+ 张）
- **新增家族** = 7 张贴图（一个家族一套）
- **新增章节** = 0 张贴图（复用已有家族）

### 9.3 迁移策略

现有 100+ 张贴图可以映射到家族贴图：

| 现有贴图 | 映射到家族贴图 |

| frost_imp.png | ice_swarm.png |
| lava_lizard.png | fire_swarm.png |
| grave_rat.png | undead_swarm.png |

首次迁移可以直接重命名/软链接，不需要重新生成。

### 10.2 分步迁移（避免大爆炸重构）

#### 第一步：新增 MonsterFamilies + BossArchetypes 模块（无破坏性）

- 创建 `MonsterFamilies.lua`，定义 8 个家族
- 添加 `Get(familyId)` 和 `Resolve(familyId, behaviorId, chapter, tagLevels)` 函数
- 创建 `BossArchetypes.lua`，定义 7 个原型 + 元素注入函数
- 添加 `Resolve(bossConfig, chapter)` 函数（输入原型三元组，输出 Spawner 兼容定义）
- 不修改任何现有代码，纯新增

#### 第二步：StageConfig 双轨运行

- 在 `StageConfig.CHAPTERS` 中新增 `families` 和 `tagLevels` 字段
- 修改 `GetStage()` 支持两种模式：
  - 如果 `stages` 存在 → 走旧逻辑（手写 waves）
  - 如果 `families` 存在且无 `stages` → 走自动编排
- 逐章迁移：每次迁移一章，删除其 `stages` 字段，添加 `families` 字段
- 回归测试每章

#### 第三步：清理

- 删除 `StageConfig.MONSTERS` 中已被家族覆盖的怪物定义
- 删除旧的 `stages` 手写配置
- 保留 Boss 定义在 `StageConfig.BOSSES` 中

### 10.3 验证方法

每迁移一章后，验证以下内容一致：

1. 每关怪物种类和数量（允许小幅调整）
2. 怪物 HP/ATK/DEF 在相同 scaleMul 下的值
3. 怪物抗性值
4. 怪物能力标签（减速/穿甲/爆炸等）
5. Boss 行为不受影响

## 十一、扩展性

### 11.1 新增章节（改造后）

```
1. 选择 1-3 个已有家族
2. 选择 Boss 原型 (中Boss 4选1, 终Boss 3选1)
3. 指定元素 → Boss 自动获得对应 Debuff 风味
4. 填写章节配置 (~15行，含 families + tagLevels + boss 三元组)
5. 完成 ✓
```

**不需要**: 写怪物定义、写 waves 配置、画怪物贴图、设计 Boss 技能组合。
**总配置量**: ~15 行（章节表） + 0 美术成本。

### 11.2 新增家族

```
1. 定义家族 (~40行): 7 个成员的 name/image/resistId
2. 生成 7 张贴图
3. 在任意章节的 families 列表中引用
4. 完成 ✓
```

### 11.3 为 IP 系统铺路

家族系统引入了"怪物等级"的概念（通过章节和 scaleMul），这直接对应 IP 系统中"掉落物品的 Item Power 由怪物等级决定"：

```
怪物等级 = f(chapter, stage)
掉落装备 IP = 怪物等级 × 随机浮动
```

## 十二、风险与注意事项

| 风险 | 缓解措施 |
|------|---------|
| 自动编排的关卡感觉"千篇一律" | 编排模板提供 10 种不同模式；支持 `waveOverrides` 手动调整个别关卡 |
| 怪物数量减少导致视觉重复 | 56 张家族贴图 vs 当前 100+，但每张贴图出场频率更高——如果玩家反馈重复，可以加"章节色调偏移"机制 |
| Boss 原型战斗体验同质化 | 元素注入系统让同原型 Boss 有完全不同的 Debuff 风味（冰=减速地狱，毒=防御崩塌，火=攻速压制）；168 种组合远超需求量 |
| 折光矿脉/无尽试炼的怪物适配 | ResourceDungeonConfig 有独立配置不受影响；EndlessTrial 使用 MonsterTemplates.Assemble 已兼容 |
| 旧版 Ch1-12 Boss 迁移兼容 | BossSkillTemplates.Update 已有路由：`phases` 走模板系统，无 `phases` 走旧逻辑，可逐章迁移 |
