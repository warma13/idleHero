# 多槽位云端存档系统设计

适用于 WASM/移动端场景的云端优先、本地缓存、分片存储的多槽位存档方案。

---

## 1. 设计背景与约束

| 约束 | 说明 |
|------|------|
| WASM 无持久存储 | 每次启动本地存储为空，不能依赖本地文件做主存储 |
| 云变量单 key 大小限制 | 单个 key 的 value 不能超过约 10KB |
| 存档数据会持续膨胀 | 随游戏进度增加，总数据量可能达到几十 KB |
| 需要多存档槽位 | 玩家可管理多个独立存档 |
| 需要离线时长检测 | 加载时能知道玩家离开了多久 |

---

## 2. 整体架构

```
┌────────────────────────────────────┐
│         业务层 (游戏逻辑)           │
│  Save() / SaveNow() / MarkDirty() │
└──────────────┬─────────────────────┘
               │
               ▼
┌────────────────────────────────────┐
│       SlotSaveSystem (唯一接口)     │
│  序列化 · 分片 · 云端读写 · 迁移    │
└───────┬──────────────┬─────────────┘
        │              │
        ▼              ▼
   ┌─────────┐  ┌────────────┐
   │本地文件  │  │ 云变量 API  │
   │(同步写)  │  │ (异步读写)  │
   └─────────┘  └────────────┘
```

**核心原则**：
- **云端是权威数据源**，启动只从云端加载
- **本地是运行时缓存**，先同步写本地保证不丢，再异步上传云端
- **SlotSaveSystem 是唯一入口**，业务层不直接接触云变量或本地文件

## 3. 云端 Key 布局

```
cloud keys:
├── save_meta                # 全局概要 (所有槽位的摘要信息)
│
├── s_1_head                 # 槽位1 分片头 (索引 + 校验)
├── s_1_groupA               # 槽位1 数据组A
├── s_1_groupB               # 槽位1 数据组B
├── s_1_groupC               # 槽位1 数据组C
├── s_1_groupC_0             # (若 groupC 超过单 key 上限, 拆为多片)
├── s_1_groupC_1
│
├── s_2_head                 # 槽位2 (同上结构)
├── s_2_groupA
│ ...
└── s_N_...                  # 槽位N
```

### 3.1 save_meta

存储所有槽位的**摘要信息**，用于存档选择界面快速展示，不需要加载完整存档。

```lua
save_meta = {
    version    = 1,
    activeSlot = 2,          -- 上次使用的槽位
    slots = {
        ["1"] = {            -- 每个槽位的摘要 (由业务层定义展示哪些字段)
            label1    = ...,
            label2    = ...,
            playTime  = 36000,
            timestamp = 1710590400,
            createdAt = 1710000000,
        },
        ["2"] = { ... },
    },
}
```

设计要点：
- meta 中**只放展示用的摘要**，不放完整游戏数据
- 每次保存时同步更新 meta（仅当摘要有变化时才写入，做 snapshot 比对优化）
- meta 本身足够小，不需要分片

### 3.2 分片 head

每个槽位有一个 head key，记录该槽位所有数据组的**索引和校验信息**：

```lua
s_N_head = {
    format    = 2,               -- 分片格式版本号
    version   = 6,               -- 存档数据版本号
    timestamp = 1710590400,
    slotId    = N,
    keys = {
        groupA = {               -- 单片: 校验码 + 长度
            cs  = 123456,
            len = 2048,
        },
        groupB = {               -- 多片: 分片数 + 每片校验码 + 每片长度
            chunks = 3,
            cs     = { 111, 222, 333 },
            len    = { 9000, 9000, 2000 },
        },
    },
}
```

## 4. 数据分组

将完整存档数据拆分为**多个功能组**，每组独立编码为 JSON 并存到单独的 key。

### 4.1 为什么要分组

- 单 key 有大小限制（如 10KB），完整存档可能远超此限制
- 不同功能域的数据增长速率不同（背包会膨胀，基础信息相对稳定）
- 分组后可独立编解码，单组损坏不影响其他组

### 4.2 分组方式

业务层定义 `SplitIntoGroups(saveData)` 函数，将序列化后的完整数据拆分为若干命名组：

```lua
-- 由业务层按自身需求定义分组
local function SplitIntoGroups(saveData)
    return {
        core     = { ... },   -- 核心进度 (等级/关卡等, 通常较小)
        currency = { ... },   -- 货币类数据
        items    = { ... },   -- 物品/装备 (可能较大, 会被进一步分片)
        skills   = { ... },   -- 技能/天赋
        misc     = { ... },   -- 杂项配置
    }
end
```

对应的 `MergeGroups(groups)` 将各组合并还原为完整 saveData。

### 4.3 超大组的自动分片

单个组编码后若仍超过 key 大小上限，自动按字节拆分为多个分片：

```
groupC (20KB JSON)
  → s_N_groupC_0 (9KB)
  → s_N_groupC_1 (9KB)
  → s_N_groupC_2 (2KB)
```

分片信息记录在 head 的 `keys.groupC.chunks` 中，读取时按顺序拼接还原。

## 5. 数据压缩策略

对于重复结构多、字段名冗长的数据（如装备列表），在序列化时进行压缩以节省空间：

### 5.1 短 key 映射

将可读的长字段名替换为 1-2 字符的短名：

```lua
-- 写入时: { slot = "weapon", qualityIdx = 3 } → { s = "weapon", q = 3 }
-- 读取时: 反向还原
```

### 5.2 去除可推导字段

从 Config 可推导的冗余字段不存入存档，读取时动态补回：

```lua
-- 不存: qualityName, qualityColor (可从 Config[qualityIdx] 查到)
-- 读取时: item.qualityName = Config.QUALITY[item.qualityIdx].name
```

### 5.3 稀疏写入

默认值/零值的字段不写入，读取时用默认值填充：

```lua
-- locked = false 时不写入 lk 字段
-- upgradeLv = 0 时不写入强化相关字段
```

## 6. 校验机制

### DJB2 校验码

每个分片在编码时计算 DJB2 哈希，写入 head。读取时重新计算并比对：

```lua
local function CalcChecksum(str)
    local hash = 5381
    for i = 1, #str do
        hash = ((hash << 5) + hash + string.byte(str, i)) & 0xFFFFFFFF
    end
    return hash
end
```

校验失败时的处理：不直接报错，而是回退到备选格式（见容错章节）。

## 7. 生命周期流程

### 7.1 初始化 (Init)

```
Init(onMetaReady)
    │
    ├─ 从云端读取 save_meta
    │
    ├─ 【有 meta】→ 回调 (meta, isNewPlayer=false)
    │
    ├─ 【无 meta】→ 检查旧格式 key，执行迁移
    │   ├─ 有旧存档 → 迁移到新槽位，创建 meta
    │   └─ 无旧存档 → 新玩家，创建空 meta
    │
    └─ 【网络错误】→ 指数退避重试，全部失败则回调错误
```

### 7.2 加载存档 (LoadSlot)

```
LoadSlot(slotId, onComplete)
    │
    ├─ 读取 s_N_head（同时读旧格式 key 作为 fallback）
    │
    ├─ 【分片格式】
    │   ├─ 从 head.keys 收集所有分组 key
    │   ├─ 批量读取所有分组
    │   ├─ 每组: 拼接分片 → 校验 DJB2 → JSON 解码
    │   └─ 合并所有组 → 完整 saveData
    │
    ├─ 【旧格式 / 分片失败】→ 回退读取单 key 数据
    │
    └─ FinalizeLoad:
        ├─ 版本迁移（逐级升级）
        ├─ 数据解压（短 key → 长 key，补回可推导字段）
        ├─ 反序列化到运行时状态
        ├─ 计算离线时长 = now - saveData.timestamp
        ├─ 设置 saveConfirmed = true，开始自动保存循环
        └─ 回调 onComplete(true)
```

### 7.3 保存 (Save / SaveNow / MarkDirty)

三种触发方式：

| 方式 | 场景 | 行为 |
|------|------|------|
| `Save()` | 自动保存定时器触发 | 序列化 → 本地 → 云端 |
| `SaveNow()` | 关键事件（通关、购买等） | 同 Save()，但立即执行、重置定时器 |
| `MarkDirty()` | 频繁的小改动 | 设置延迟定时器，N 秒后合并为一次 Save() |

```
Save / SaveNow:
    │
    ├─ Serialize() → 收集运行时状态为 saveData
    │
    ├─ SaveLocal(saveData) → 原子写入本地文件
    │   └─ 先写 .tmp → rename（防写入中断损坏）
    │
    ├─ SplitIntoGroups(saveData) → 拆分为功能组
    │
    ├─ EncodeChunkedGroups(slotId, groups):
    │   └─ 每组: JSON 编码 → 按大小分片 → 计算校验码
    │   └─ 生成 kvPairs + headData
    │
    └─ 异步云端批量写入 (BatchSet):
        ├─ head + 所有分组 key
        ├─ 更新 save_meta
        ├─ 清理旧分片残留（若分片数量变化）
        ├─ 可选: 兼容回写旧格式 key
        ├─ 成功 → 更新 head 缓存
        └─ 失败 → 后台重试（最多 N 次，间隔递增）
```

### 7.4 新建存档 (CreateNewSlot)

```
CreateNewSlot(slotId, onComplete):
    ├─ 重置运行时状态到初始值
    ├─ 序列化 → 本地保存 → 云端保存
    ├─ 更新 save_meta（添加新槽位摘要）
    └─ 设置 saveConfirmed = true
```

### 7.5 删除存档 (DeleteSlot)

```
DeleteSlot(slotId, onComplete):
    ├─ 校验: 不能删除当前活跃存档
    ├─ 从 save_meta 移除该槽位
    ├─ 云端批量删除:
    │   ├─ 优先从缓存的 head 获取精确 key 列表
    │   └─ 无缓存则保守删除所有可能的 key（含分片后缀）
    └─ 更新 save_meta
```

### 7.6 另存为 (CopyToSlot)

```
CopyToSlot(targetSlot, onComplete):
    ├─ 序列化当前进度
    ├─ 分片编码 → 写入目标槽位
    ├─ 清理目标槽位旧分片残留
    ├─ 更新 save_meta
    └─ 不切换活跃槽位（仅复制）
```

### 7.7 切换存档 (SaveAndUnload)

```
SaveAndUnload(onComplete):
    ├─ SaveNow() 保存当前进度
    ├─ 重置内部状态 (saveConfirmed = false, 清空计时器)
    ├─ 重新从云端拉取最新 save_meta
    └─ 回调 → UI 重新展示存档选择界面
```

## 8. 自动保存循环

在每帧 Update 中管理：

```
Update(dt):
    ├─ 延迟重试计时器（Init 阶段）
    │
    ├─ 累计游戏时长 playTime += dt
    │
    ├─ 云端重试计时器（保存失败后的重试）
    │
    ├─ 自动保存计时器
    │   └─ 每 SAVE_INTERVAL 秒 → 触发 Save()
    │
    └─ 脏标记计时器
        └─ MarkDirty() 后 DIRTY_DELAY 秒 → 触发 Save()
        └─ 与自动保存合并，避免重复保存
```

## 9. 版本迁移框架

存档数据包含 `version` 字段。加载时通过**迁移链**逐级升级：

```lua
local MIGRATIONS = {
    [1] = function(data) ... end,   -- v1 → v2
    [2] = function(data) ... end,   -- v2 → v3
    [3] = function(data) ... end,   -- v3 → v4
}

local function RunVersionMigrations(data)
    while data.version < CURRENT_VERSION do
        local migrateFn = MIGRATIONS[data.version]
        if migrateFn then
            migrateFn(data)
            data.version = data.version + 1
        else
            break  -- 无对应迁移函数, 中断
        end
    end
end
```

迁移函数的职责：为新版本添加缺失字段的默认值、转换数据结构等。

## 10. 旧格式迁移

从单存档系统升级到多槽位系统时，需要一次性迁移：

```
RunMigration:
    ├─ 读取旧 key（如 save_data, manual_save）
    │
    ├─ 有旧数据 → 分片编码后写入对应槽位
    │   ├─ 旧自动存档 → 槽位 1
    │   └─ 旧手动存档 → 槽位 2
    │   └─ 标记 migratedFrom 来源
    │
    ├─ 创建 save_meta
    │
    └─ 无旧数据 → 新玩家，创建空 meta
```

迁移后保留旧 key 不删除，以防万一需要回滚。

## 11. 容错策略

| 场景 | 处理 |
|------|------|
| 云端读取失败 | 指数退避重试（如 3s, 9s, 27s），最多 N 次 |
| 分片校验不匹配 | 回退到旧格式单 key 读取 |
| 分片解码失败 | 回退到旧格式单 key 读取 |
| 云端保存失败 | 本地已保存不阻塞游戏，后台定时重试 |
| 新建存档云端失败 | 本地已保存，继续游戏 |
| 本地写入失败 | 仅打印警告，依赖云端 |
| 数据格式不确定 | 运行时检测字段判断新旧格式，自适应解压 |

关键思想：**永远不因存档问题阻塞游戏进程**。本地保底 → 云端异步 → 失败重试 → 旧格式兜底。

## 12. 接口设计

```lua
-- 初始化 (启动时调用一次)
SlotSaveSystem.Init(onMetaReady)
-- onMetaReady(meta, isNewPlayer, err)

-- 存档操作
SlotSaveSystem.LoadSlot(slotId, onComplete)
SlotSaveSystem.CreateNewSlot(slotId, onComplete)
SlotSaveSystem.DeleteSlot(slotId, onComplete)
SlotSaveSystem.CopyToSlot(targetSlot, onComplete)
SlotSaveSystem.SaveAndUnload(onComplete)

-- 保存触发
SlotSaveSystem.Save()        -- 常规保存
SlotSaveSystem.SaveNow()     -- 立即保存
SlotSaveSystem.MarkDirty()   -- 标记脏数据, 延迟合并

-- 每帧调用
SlotSaveSystem.Update(dt)

-- 查询
SlotSaveSystem.GetMeta()          -- 返回 save_meta
SlotSaveSystem.GetActiveSlot()    -- 当前槽位号 (0=未选择)
SlotSaveSystem.GetSlotCount()     -- 已用槽位数
SlotSaveSystem.GetMaxSlots()      -- 最大槽位数
SlotSaveSystem.GetPlayTime()      -- 累计游戏时长
SlotSaveSystem.IsSaveHealthy()    -- 存档健康状态
```

### 业务层需实现的扩展点

| 扩展点 | 说明 |
|--------|------|
| `SplitIntoGroups(saveData)` | 定义分组策略，将完整数据拆为命名组 |
| `MergeGroups(groups)` | 将各组合并还原为完整数据 |
| `Serialize()` | 从运行时状态收集数据 |
| `Deserialize(data)` | 将数据写入运行时状态 |
| `BuildMetaSlot()` | 构建当前槽位的摘要信息 |
| `MIGRATIONS[version]` | 版本迁移函数 |
| `Compress/Decompress` | 可选的数据压缩/解压 |

## 13. 关键设计决策总结

| 决策 | 理由 |
|------|------|
| 云端优先，本地缓存 | WASM 每次启动本地为空，必须以云端为权威 |
| 先写本地，再异步上云 | 保证运行中不丢数据，云端失败不阻塞游戏 |
| 数据分组 + 自动分片 | 绕过单 key 大小限制，且各组独立互不影响 |
| DJB2 校验 + 旧格式回退 | 分片损坏时有兜底方案，不丢数据 |
| 兼容回写旧 key | 过渡期双写，便于回滚 |
| 迁移链逐级升级 | 任意旧版本存档都能加载，不跳版本 |
| 原子本地写入 (.tmp + rename) | 防止写入中断导致文件损坏 |
| meta + head 两层索引 | meta 用于快速展示列表，head 用于精确加载和校验 |
| MarkDirty 合并机制 | 高频小改动不触发高频保存，延迟合并后一次性写入 |
