# 存档系统流程分析

> SlotSaveSystem.lua 的完整工作流程。
> 最后更新: 2026-03-24

---

## 1. 架构概览

```
┌──────────────────────────────────────┐
│         业务层 (各模块)               │
│  Save() / SaveNow() / MarkDirty()   │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│     SlotSaveSystem (唯一入口)         │
│  序列化 · 分片 · 云端读写 · 版本迁移  │
└───────┬──────────────┬───────────────┘
        │              │
        ▼              ▼
   ┌─────────┐  ┌────────────┐
   │本地文件  │  │ clientCloud │
   │(同步写)  │  │ (异步读写)  │
   └─────────┘  └────────────┘
```

**核心原则**：
- **云端是权威数据源**，启动只从云端加载（WASM 每次启动本地为空）
- **本地是运行时缓存**，先同步写本地保证不丢，再异步上传云端
- **SlotSaveSystem 是唯一入口**，业务层不直接接触云变量或本地文件

## 2. 核心常量

| 常量 | 值 | 说明 |
|------|----|------|
| `MAX_SLOTS` | 10 | 最大存档槽位数 |
| `SAVE_FORMAT` | 2 | 分片存档格式版本 |
| `CURRENT_SAVE_VERSION` | 6 | 当前存档数据版本 |
| `SAVE_INTERVAL` | 30 秒 | 自动保存间隔 |
| `DIRTY_DELAY` | 3 秒 | 脏标记延迟保存 |
| `CHUNK_MAX_BYTES` | 9 KB | 单个云变量 key 最大字节数（留 1KB 安全余量） |
| `MAX_RETRY` | 3 | 云端请求最大重试次数 |
| `MIN_OFFLINE_SEC` | 300 秒 (5 分钟) | 触发离线奖励的最短离线时间 |
| `MAX_OFFLINE_SEC` | 28800 秒 (8 小时) | 离线奖励计算的上限时间 |

## 3. 云端 Key 布局

```
clientCloud Keys:
├── save_meta              # 所有槽位的概要（快速展示存档列表）
├── save_data              # 兼容回写（旧格式完整存档，每次保存同步更新）
│
├── s_1_head               # 槽位 1: 分片索引（格式版本 + 校验码）
├── s_1_core               # 槽位 1: 核心数据组
├── s_1_currency           # 槽位 1: 货币数据组
├── s_1_equip              # 槽位 1: 装备数据组（压缩）
├── s_1_inv                # 槽位 1: 背包数据组（压缩，超限时拆为 _0, _1, ...）
├── s_1_skills             # 槽位 1: 技能数据组
├── s_1_misc               # 槽位 1: 杂项数据组（域注册字段汇入此处）
│
├── s_2_head               # 槽位 2（同上结构）
│ ...
└── s_10_...               # 槽位 10
```

## 4. 两层索引结构

### 4.1 save_meta（全局概要）

用于存档选择界面快速展示，**不包含游戏数据**：

```lua
save_meta = {
    version    = 1,
    activeSlot = 3,            -- 上次使用的槽位号 (1-10)
    slots = {
        ["1"] = {
            level     = 50,
            chapter   = 5,
            stage     = 8,
            maxFloor  = 120,
            playTime  = 36000,     -- 累计游戏时长(秒)
            saveCount = 150,
            timestamp = 1710590400,
        },
        -- 空槽位不写入
    },
}
```

**优化**：维护 `lastMetaSnapshot_`，仅当展示字段变化时才写入 meta。

### 4.2 s_N_head（槽位分片索引）

记录该槽位所有数据组的校验和分片信息：

```lua
s_N_head = {
    format    = 2,              -- 分片格式版本
    version   = 6,              -- 存档数据版本
    timestamp = 1710590400,
    slotId    = N,
    keys = {
        core     = { cs = 123456, len = 2048 },                       -- 单分片
        inv      = { chunks = 2, cs = {111, 222}, len = {9000, 3000} }, -- 多分片
        -- ...
    },
}
```

## 5. 数据分组与分片

### 5.1 分组策略

完整存档数据通过 `SplitIntoGroups()` 拆分为 **6 个功能组**，每组独立编码为 JSON，存到单独的云变量 key：

| 组名 | 内容 |
|------|------|
| **core** | 版本、时间戳、元信息、玩家基础、关卡进度、记录 |
| **currency** | 材料、扩容计数 |
| **equip** | 装备栏（压缩后） |
| **inv** | 背包列表（压缩后，增长最快，可能触发多分片） |
| **skills** | 技能树、药水 Buff |
| **misc** | 硬编码杂项 + 域注册字段（见 §6） |

加载时通过 `MergeGroups()` 将各组合并还原为完整 saveData。

### 5.2 自动分片

单个组编码后若超过 `CHUNK_MAX_BYTES`（9KB），按字节切分为多个分片：

```
inv (20KB JSON)
  → s_N_inv_0 (9KB)
  → s_N_inv_1 (9KB)
  → s_N_inv_2 (2KB)
```

分片信息记录在 head 的 `keys.inv.chunks` 中，读取时按序拼接还原。

### 5.3 旧分片清理

每次保存时，与上次缓存的 `lastSavedHead_` 对比。如果某组的分片数量减少（如 inv 从 3 片变为 2 片），多出的旧 key 在同一次 BatchSet 中 Delete 掉。

## 6. 域注册机制 (RegisterDomain)

各业务模块通过 `RegisterDomain()` 自注册序列化/反序列化逻辑，**消除 SlotSaveSystem 对具体字段的硬编码依赖**：

```lua
SlotSaveSystem.RegisterDomain({
    name  = "moduleName",           -- 域名称
    keys  = { "field1", "field2" }, -- 拥有的字段名列表
    group = "misc",                 -- 存入哪个分组
    serialize   = function(GS) return { field1 = GS.field1 } end,
    deserialize = function(GS, data) GS.field1 = data.field1 or default end,
})
```

**已注册的域**：endlessTrial, worldBoss, resourceDungeon, dailyRewards, titles, forge, abyss

**工作方式**：
- `Serialize()` 循环调用所有域的 `serialize` 回调，合并结果到 saveData
- `Deserialize()` 循环调用所有域的 `deserialize` 回调
- `SplitIntoGroups()` 根据域声明的 `group` 将字段注入到对应分组

## 7. 数据压缩策略

### 7.1 短 key 映射

写入时将长字段名替换为 1-2 字符短名，读取时反向还原。

### 7.2 去除可推导字段

从 Config 可推导的冗余字段不存入存档，读取时动态补回。

### 7.3 稀疏写入

默认值/零值的字段不写入，读取时用默认值填充。

### 7.4 格式自检测

解压时通过检测 `c.slot`（完整格式）vs `c.s`（压缩格式）自动判断是否需要解压，兼容新旧两种格式。

## 8. 校验机制

每个分片编码时计算 **DJB2 哈希**（32 位无符号），存入 head 的 `keys[group].cs`。读取时重新计算并比对，不匹配则回退到旧格式单 key 读取。

```lua
local function CalcChecksum(str)
    local hash = 5381
    for i = 1, #str do
        hash = ((hash << 5) + hash + string.byte(str, i)) & 0xFFFFFFFF
    end
    return hash
end
```

## 9. 完整流程

### 9.1 初始化 (Init)

```
Init(onMetaReady)
    │
    ├─ 从云端读取 "save_meta"
    │
    ├─ 【有 meta】
    │   └─ 回调 onMetaReady(meta, isNewPlayer=false)
    │       → UI 展示存档选择界面
    │
    ├─ 【无 meta】→ RunMigration
    │   ├─ 读取旧 key: "save_data" + "manual_save"
    │   ├─ 有旧存档 → 迁移到槽位 1/2，创建 save_meta
    │   └─ 无旧存档 → 新玩家，创建空 save_meta
    │
    └─ 【网络错误】→ 指数退避重试 (3s, 9s, 27s)
        └─ 全部失败 → 回调错误信息
```

### 9.2 加载存档 (LoadSlot)

```
LoadSlot(slotId, onComplete)
    │
    ▼
BatchGet: s_N_head + save_slot_N (旧格式兼容)
    │
    ├─ 【head 存在 & format=2】→ 分片格式
    │   ├─ 从 head.keys 收集所有分组 key
    │   ├─ BatchGet 读取所有分组 key
    │   ├─ DecodeChunkedGroups:
    │   │   每组: 拼接分片 → 校验 DJB2 → JSON decode
    │   └─ MergeGroups → 完整 saveData
    │
    └─ 【无 head / 分片失败】→ 回退旧格式
        └─ 直接使用 save_slot_N 的 table 数据

        ↓ (两条路径汇合)

FinalizeLoad(slotId, saveData):
    ├─ ValidateStructure() → 字段完整性校验
    ├─ 解压装备数据（自动检测格式）
    ├─ MigrateData() → 版本迁移链 (v1→v2→...→v6)
    ├─ Deserialize(saveData) → 写入运行时状态
    │   ├─ 硬编码字段（player/equipment/inventory/...）
    │   └─ 域注册字段（循环调用各域的 deserialize）
    ├─ 计算离线时长 → offlineSeconds
    ├─ 设置 saveConfirmed_ = true，开启自动保存
    └─ 回调 onComplete(true)
```

### 9.3 保存 (Save / SaveNow / MarkDirty)

```
触发方式:
  ├─ 自动保存: Update() 中 30 秒定时触发
  ├─ MarkDirty(): 标记脏数据, 3 秒后合并保存
  └─ SaveNow(): 立即保存（通关、购买等关键事件）

Save():
    │
    ├─ Serialize()
    │   ├─ 收集硬编码字段
    │   └─ 循环调用域注册的 serialize 回调
    │
    ├─ SaveLocal(saveData) → 原子写本地（.tmp → rename）
    │
    └─ DoCloudSave(saveData):
        ├─ SplitIntoGroups → 拆分为 6 组
        ├─ EncodeChunkedGroups → 分片 + 校验码
        ├─ 一次 BatchSet 原子写入:
        │   ├─ head + 所有分组 key
        │   ├─ 清理旧分片残留
        │   ├─ save_meta（仅摘要变化时写入）
        │   ├─ save_data（兼容回写）
        │   └─ iscores 排行榜字段
        ├─ 成功 → 缓存 head，重置重试
        └─ 失败 → 指数退避重试（最多 3 次）
```

### 9.4 新建存档 (CreateNewSlot)

```
CreateNewSlot(slotId, onComplete):
    ├─ 重置运行时状态到初始值
    ├─ Serialize() → SaveLocal() → 云端 BatchSet
    ├─ 更新 save_meta（添加新槽位摘要）
    └─ 设置 saveConfirmed_ = true
```

### 9.5 删除存档 (DeleteSlot)

```
DeleteSlot(slotId, onComplete):
    ├─ 校验: 不能删除当前活跃存档
    ├─ 从 saveMeta_.slots 移除该槽位
    ├─ BatchSet:
    │   ├─ Delete: 旧格式 key + head + 所有分组 key
    │   │   优先从缓存 head 获取精确 key 列表
    │   │   无缓存则保守删除所有可能 key（含 _0~_9 分片后缀）
    │   └─ Set: 更新后的 save_meta
    └─ 清除 head 缓存
```

### 9.6 另存为 (CopyToSlot)

```
CopyToSlot(targetSlot, onComplete):
    ├─ Serialize() 当前进度（_meta.slotId 改为目标槽位）
    ├─ 分片编码 → BatchSet 写入目标槽位
    ├─ 清理目标槽位旧分片残留
    ├─ 更新 save_meta
    └─ 不切换 currentSlot_（仅复制）
```

### 9.7 切换存档 (SaveAndUnload)

```
SaveAndUnload(onComplete):
    ├─ SaveNow() 保存当前进度
    ├─ 重置内部状态: saveConfirmed_=false, currentSlot_=0, 清空计时器
    ├─ 重新从云端拉取最新 save_meta
    └─ 回调 → UI 重新展示存档选择界面
```

## 10. 自动保存循环 (Update)

`SlotSaveSystem.Update(dt)` 每帧调用，管理 5 个计时器：

```
Update(dt):
    │
    ├─ 1. 延迟重试计时器（Init/Migration 阶段的网络重试）
    │
    ├─ 2. playTime_ += dt（累计游戏时长）
    │
    ├─ 3. 云端重试计时器（保存失败后的自动重试）
    │
    ├─ 4. 自动保存计时器 → 每 SAVE_INTERVAL 秒触发 Save()
    │
    └─ 5. 脏标记计时器 → MarkDirty() 后 DIRTY_DELAY 秒触发 Save()
                         与自动保存互斥，触发后重置自动保存计时器
```

## 11. 版本迁移链

存档数据包含 `version` 字段，加载时逐级执行迁移函数。迁移前自动备份原始数据到本地文件。

| 版本 | 迁移内容 |
|------|---------|
| v1 → v2 | elemDmg 词条值缩减为 1/5 |
| v2 → v3 | 技能 ID 调和 + 装备字段规范化 + 材料字段初始化 |
| v3 → v4 | 缩放公式从指数改为对数，装备非百分比词条按比例缩放 |
| v4 → v5 | 经济重平衡（占位迁移） |
| v5 → v6 | 删除已移除套装的装备，分解补偿为强化石 |

当前版本: **CURRENT_SAVE_VERSION = 6**

## 12. 旧存档迁移 (RunMigration)

首次使用多槽位系统时（save_meta 不存在），自动迁移旧格式存档：

| 旧 key | 迁移目标 | 标记 |
|--------|---------|------|
| `save_data`（自动存档） | 槽位 1 | `migratedFrom = "auto_save"` |
| `manual_save`（手动存档） | 槽位 2 | `migratedFrom = "manual_save"` |

- 两份存档**不合并**，分别放入不同槽位，让用户自行选择
- 迁移写入使用单次 BatchSet 保证原子性
- 迁移失败时指数退避重试，全部失败则阻止进入游戏（防止空数据覆盖）
- 迁移后旧 key 不删除，保留作为最后数据恢复手段

## 13. 离线时长检测

加载存档时，通过 `timestamp` 计算离线时长：

```
elapsed = os.time() - savedTimestamp
offlineSeconds = elapsed >= MIN_OFFLINE → clamp(elapsed, 0, MAX_OFFLINE) → 交给业务层处理
                 elapsed < MIN_OFFLINE  → 0（不触发离线奖励）
```

## 14. 容错策略

| 场景 | 处理方式 |
|------|---------|
| 云端读取失败 | 指数退避重试 (3s, 9s, 27s)，最多 3 次 |
| 分片校验码不匹配 | 回退到旧格式 `save_slot_N` 单 key 读取 |
| 分片解码失败 | 回退到旧格式 `save_slot_N` 单 key 读取 |
| 云端保存失败 | 本地已保存，不阻塞游戏；后台定时重试 |
| 新建存档云端失败 | 本地已保存，继续游戏 |
| 版本字段缺失 | 迁移链自动补全 |
| 数据压缩格式不确定 | 运行时自动检测新旧格式，自适应解压 |

**核心思想**：永远不因存档问题阻塞游戏。本地保底 → 云端异步 → 失败重试 → 旧格式兜底。

## 15. 公开 API

```lua
-- 初始化（启动时调用一次）
SlotSaveSystem.Init(onMetaReady)
-- onMetaReady(meta, isNewPlayer, err)

-- 存档操作
SlotSaveSystem.LoadSlot(slotId, onComplete)
SlotSaveSystem.CreateNewSlot(slotId, onComplete)
SlotSaveSystem.DeleteSlot(slotId, onComplete)
SlotSaveSystem.CopyToSlot(targetSlot, onComplete)
SlotSaveSystem.SaveAndUnload(onComplete)

-- 保存触发
SlotSaveSystem.Save()         -- 常规保存（本地 + 云端）
SlotSaveSystem.SaveNow()      -- 立即保存（重置定时器）
SlotSaveSystem.MarkDirty()    -- 脏标记，延迟合并

-- 每帧调用
SlotSaveSystem.Update(dt)

-- 域注册（各模块在自身文件末尾调用）
SlotSaveSystem.RegisterDomain({ name, keys, group, serialize, deserialize })

-- 查询
SlotSaveSystem.GetMeta()          -- 返回 save_meta
SlotSaveSystem.GetActiveSlot()    -- 当前槽位号 (0=未选择)
SlotSaveSystem.GetSlotCount()     -- 已用槽位数
SlotSaveSystem.GetMaxSlots()      -- 最大槽位数 (10)
SlotSaveSystem.GetPlayTime()      -- 累计游戏时长(秒)
SlotSaveSystem.IsSaveHealthy()    -- 存档健康状态
```

## 16. 关键设计决策

| 决策 | 理由 |
|------|------|
| 云端优先，本地缓存 | WASM 每次启动本地为空，必须以云端为权威 |
| 先写本地，再异步上云 | 保证运行中不丢数据，云端失败不阻塞游戏 |
| 数据分组 + 自动分片 | 绕过单 key 大小限制，各组独立互不影响 |
| 域注册 (RegisterDomain) | 新增存档字段时无需修改 SlotSaveSystem，各模块自治 |
| DJB2 校验 + 旧格式回退 | 分片损坏时有兜底方案，不丢数据 |
| 兼容回写旧 save_data | 过渡期双写，便于版本回滚 |
| 迁移链逐级升级 | 任意旧版本存档都能加载，不跳版本 |
| 原子本地写入 (.tmp + rename) | 防止写入中断导致文件损坏 |
| meta + head 两层索引 | meta 快速展示列表，head 精确加载和校验 |
| meta 写入优化 (snapshot 比对) | 减少约 60-80% 的无变化 meta 写入 |
| MarkDirty 合并机制 | 高频小改动不触发高频保存，延迟合并一次写入 |
